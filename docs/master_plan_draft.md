## 1. Core Security Axioms & Invariants Reference

Every component of the Kaia architecture must strictly enforce and maintain the following nonвҖ‘negotiable invariants. Any implementation path that introduces a bypass to these rules represents a critical vulnerability.

### 1.1 The NonвҖ‘Negotiable Axioms

1. **LLM Outputs are Untrusted Payloads:** All actions, script proposals, parameter assignments, and queries generated by a Large Language Model (LLM) must be handled as untrusted userвҖ‘space network input. They must never be allowed direct or unvalidated access to the host shell or privileged APIs.

2. **Deterministic Schema Validation:** All administrative or operations intents must be parsed and strictly validated against strongly typed Pydantic models by a deterministic host process before being passed to an execution layer.

3. **HostвҖ‘Retained Execution Control:** The language model operates purely as a reasoning and proposal mechanism. The actual execution layer, policy gatekeeper, and containment boundary remain completely deterministic, hardcoded, and separate from the model's context window or cognitive processing loop.

4. **Decoupling of Personality and Policy:** Authorization states, privilege assignments, firewall enforcement rules, and token tracking systems must remain completely decoupled from the agent's conversational personality, affinity tracking matrix, relationship tiers, or active emotional vectors.

5. **The TrustвҖ‘Building Loop (Observe вҶ’ Analyze вҶ’ Recommend вҶ’ Execute):** The system must enforce this exact sequential pipeline. **The Recommend phase is a hard engineering blockade:** no automated heuristic, statistical model, or LLMвҖ‘derived confidence score may bypass the explicit operator approval step, except for preвҖ‘approved automated responses (e.g., dropping known malicious IPs) that are themselves codified in deterministic, immutable policies. Direct `Observe вҶ’ Execute` shortвҖ‘circuiting is structurally forbidden by a system invariant.

## 3. OutвҖ‘ofвҖ‘Process Typed Policy Gate & Unix IPC Framework

To prevent memory injection exploits and maintain a resilient execution environment, the Policy Gate runs completely outвҖ‘ofвҖ‘process as a standalone daemon communicating over an isolated Unix Domain Socket.

### 3.1 IPC Sockets & Framing Protocol

- **Socket Path:** `/run/kaiacord/policy_gate.sock`
- **Group Ownership:** `kaiacord` (set via `os.chown` to root user and administrative group bounds)
- **Permissions Mask:** `0o660` (restricting standard user worldвҖ‘read/write access)

**Framing Paradigm:** Every packet starts with a 4вҖ‘byte bigвҖ‘endian unsigned integer indicating the exact byte length of the following UTFвҖ‘8 encoded JSON string payload. No newlines or sentinel bytes are used. The receiver must first read exactly 4 bytes, interpret as the length, then read that many bytes and parse as JSON.

### 3.2 IPC Data Contracts

The following Pydantic schemas must be enforced for all communications:

- **Request:** Must contain a UUID `request_id`, an `action` field restricted to one of `["diagnostics", "block_ip", "restart_service", "write_file", "run_script"]`, a `payload` dictionary with actionвҖ‘specific parameters, and a `capability_token` string matching the pattern `^ck_[A-Za-z0-9_-]+$`.
- **Response:** Must contain the same `request_id`, a boolean `approved`, an `executor_response` dictionary, an `audit_id` string, and an optional `error_message`.

All validation must be performed by the Policy Gate before any action is taken.

## 4. Advanced Local Telemetry Pipeline & Kernel Observers

### 4.1 HighвҖ‘Performance File Integrity Monitoring (FIM) via `fanotify`

The FIM subsystem must interface with the Linux kernel's native `fanotify` API using direct `ctypes` bindings (or equivalent lowвҖ‘level system calls). The implementation must:

* Initialize a fanotify file descriptor with flags `FAN_CLASS_NOTIF`, `FAN_CLOEXEC`, and `FAN_NONBLOCK`.
* Add a mark on **the entire mount point** of the monitored filesystem (e.g., the root or specific critical paths) with `FAN_MARK_ADD | FAN_MARK_MOUNT`, monitoring the events: `FAN_MODIFY`, `FAN_CREATE`, `FAN_ATTRIB`, and `FAN_ONDIR`. This provides mountвҖ‘wide coverage, capturing all file system modifications.
* In addition to fanotify, **deep eBPF hooks** must be placed on `sys_enter_openat2` and `sys_enter_unlinkat` to capture file access attempts and deletions at the syscall layer before they complete.
* For each fanotify event, consume the event file descriptor directly from the fanotify metadata structure, **read the file content synchronously via `/proc/self/fd/<fd>`** (eliminating TOCTOU races), and instantly execute cryptographic signature verification and YARA scanning on the binary headers **before the process context initialises**, blocking rootkits and payload drops in real time.
* For eBPF events on `openat2`/`unlinkat`, feed the file path into the YARA engine or signature verifier immediately.
* Log the event (timestamp, event type, PID, process comm, target path, YARA matches) to an appendвҖ‘only SQLite audit trail located outside the monitored paths (e.g., `/var/lib/secdaemon/fim_audit.db`).
* On an internal scanning error or engine crash, the telemetry thread must isolate the exception, log a critical warning to the append-only ledger, drop the associated transaction path via the Policy Gate, and trigger a graceful systemd unit restart sequence for the affected daemon.
* Exclude the following paths from monitoring to prevent feedback loops and reduce noise:
* `/var/log/*` вҖ“ exclude `FAN_MODIFY` to avoid logging echoвҖ‘loops.
* `/tmp/*`, `/var/tmp/*` вҖ“ exclude `FAN_CREATE` to avoid transient compiler thrashing.
* `/proc/*`, `/sys/*`, `/dev/*` вҖ“ complete exclusion of virtual devices.
* `/home/*/.cache/*` вҖ“ omit browser cache synchronisation.

All database operations must use a connection pool or context manager with proper error handling; any unhandled exception in the FIM thread must cause the entire daemon to terminate and restart.

### 4.2 eBPF Hook Specifications & Telemetry Filtering

The tracking engine must inject kernel probes to maintain a nonвҖ‘polling observation pipeline. **UserвҖ‘space polling (e.g., `psutil` loops) is explicitly forbidden for securityвҖ‘critical event collection; all observation must occur at the kernel level.** The following hook points and tracked metadata are required:

| Telemetry Focus | Hook Type | Probe Target | Tracked Metadata Arguments | Subsystem Objective |
| --- | --- | --- | --- | --- |
| **Process Lifecycles** | Tracepoint | `syscalls:sys_enter_execve` | `pid, uid, comm, filename, argv` | Captures nested container configurations and shells. |
| **Socket Management** | Kprobe | `kprobe:tcp_connect` | `pid, comm, daddr, dport` | Records outgoing network transaction targets. |
| **Retransmissions** | Kprobe | `kprobe:tcp_retransmit_skb` | `saddr, daddr, sport, dport, state` | Detects covert data channels. |
| **Privilege Transitions** | Tracepoint | `syscalls:sys_enter_setuid` / `sys_enter_setreuid` | `pid, comm, uid, old_uid` | Intercepts vertical escalation exploit bounds. |
| **File System Modifications** | Tracepoint | `syscalls:sys_enter_openat` / `sys_enter_openat2` | `pid, comm, filename, flags` | Monitors file access for honeypot detection and anomaly tracking; enables preвҖ‘processвҖ‘context signature checks. |
| **Unlink Operations** | Tracepoint | `syscalls:sys_enter_unlinkat` | `pid, comm, filename` | Detects deletion of critical files or honeypots. |

The eBPF programs must be written in a restricted C subset and compiled with the BCC or libbpf toolchain. On each event, the eBPF program must submit a structured record (e.g., via a perf buffer) containing the captured fields. The userspace counterpart must read these records, validate them against the defined schema, and forward them to the security event pipeline.

**Telemetry Sanitizer Rules:** Telemetry sanitization and string truncation must be executed safely on individual fields after schema validation has verified structural integrity, ensuring that truncation never invalidates the overall framing syntax of the data payload. Every incoming telemetry field (process name, IP address, hostname, etc.) must be validated against a character allowlist `[A-Za-z0-9_.-]`. All strings must be truncated to a predefined maximum length (e.g., 64 characters for hostnames, 16 for process comm). Any character outside the allowlist must be stripped or replaced with a safe placeholder to prevent injection of malicious control characters or shell metacharacters via network metadata.

## 5. Passive/Active LayerвҖ‘2 & LayerвҖ‘3 Discovery (RunZero Layer)

To preserve operational stealth and prevent exploitation paths, Kaia must track asset shifts using lowвҖ‘level socket processing loops rather than shelling out to utility scanners. **No shellвҖ‘wrapped `nmap`, `arpвҖ‘scan`, or any external binary scanner is permitted.** All parsing is done natively in Python on raw binary frames.

### 5.1 LowвҖ‘Level Raw Packet Ingestion

The discovery layer must open an unprivileged network frame interceptor bound to `AF_PACKET` with `ETH_P_ALL` on a specified network interface. It must continuously capture raw Ethernet frames and parse the headers.

### 5.2 Broadcast Protocol Extraction

The following protocols must be parsed from captured packets:

- **ARP (0x0806):** Extract sender MAC, sender IP, and operation type (request/reply). Log as an asset with detection vector `ARP`.
- **IP (0x0800) вҶ’ UDP:** For UDP packets, inspect destination port:
  - **Port 5353 (mDNS):** Parse the DNS payload to extract the queried hostname. Log asset with detection vector `mDNS`.
  - **Port 5355 (LLMNR):** Log asset with detection vector `LLMNR`.
  - **Port 1900 (SSDP):** Log asset with detection vector `SSDP`.
- **NetBIOS Name Service (UDP/137):** Parse NetBIOS name queries and responses; log asset with detection vector `NetBIOS`.

For each discovered asset, record MAC address, assigned IP, resolved hostname (if any), hardware vendor (from OUI lookup), detection vector, and timestamp. Maintain a local cache to avoid duplicate logging. This data feeds into the threat intelligence pipeline for correlation.

## 6. Proactive Defense, Tripwires, and Sandbox Mitigation

### 6.1 eBPF System Canary Core

**HoneyвҖ‘tokens** (decoy files) must be placed at sensitive paths, e.g., `/etc/api_keys.json`, `/var/backups/credentials.txt`, or similar. Additionally, **fake local configuration targets** (e.g., `~/.ssh/authorized_keys.bak`) and **dummy exposed container ports** (e.g., a mock MySQL listener on TCP/3306 inside an isolated namespace) must be deployed as decoys. The system must load an eBPF tracepoint on `sys_enter_openat` that checks if the accessed path matches any honeypot pattern. On detection, the eBPF program must submit an alert record containing PID, UID, process comm, and the accessed path. The userspace receiver must immediately:

- Log the event to `security_events.db` with severity CRITICAL.
- Trigger an automatic nftables DROP rule for the source IP (if derivable from context).
- Optionally invoke the emergency lockdown unit.

Any access to a decoy port (e.g., via a connection attempt) must similarly trigger a CRITICAL alert and immediate blocking of the source IP.

### 6.2 Network Decoy System & Netlink Firewall Architecture

An isolated network namespace (e.g., `ns_decoy_sandbox`) must host a lightweight mock listener on a set of highвҖ‘numbered ports (e.g., TCP/8080, TCP/8443) and also on commonly targeted ports (e.g., TCP/22, TCP/443) to attract scanning. The system must configure nftables to redirect or mirror traffic to these decoy ports. If any connection is made to a decoy port, the coordinator must:

- Extract the source IP from the connection metadata.
- Immediately apply an nftables DROP rule for that source IP (using raw Netlink or the `nft` command).
- Log the interception with details (source IP, timestamp, decoy port).

The decoy network must be provisioned using `ip netns` and `veth` pairs, or using `systemdвҖ‘nspawn` with a minimal container image.

### 6.3 Resource Ceilings & Containment Limits

Every application thread spawned via Bubblewrap or `systemdвҖ‘nspawn` must be bound to concrete kernel limitations via host cgroup controls:

- **CPU:** 25% system capacity (`cpu.max`)
- **Memory:** 2G maximum (`memory.max`), no swap
- **Processes:** 128 maximum (`pids.max`)
- **Execution Time:** 30 minutes per session; SIGKILL on timeout

These limits must be enforced for all script executions and userвҖ‘submitted commands that are categorised as Red or Yellow risk tiers.

### 6.4 Tamper Detection and SelfвҖ‘Protection

Kaia must continuously verify the integrity of its own critical assets to detect and respond to tampering attempts. The following **sixвҖ‘asset perimeter** must be monitored continuously:

1. **Configuration files** (e.g., `.env`, `core/config.py`, `security/schemas.py`)
2. **Executable hashes** (of `kaia_dashboard.py`, `main.py`, `security/policy_gate.py`, `security/host_executor.py`, etc.)
3. **Systemd service definitions** (unit files for Kaia components, located in `/etc/systemd/system/`)
4. **Policy Gate binary** (the Python script itself and its dependencies)
5. **Local threat intelligence databases** (InternetDB SQLite cache, future GeoLite2 MMDB, CVE databases)
6. **AppendвҖ‘only audit ledgers** (`security_events.db`, `audit_ledger.json`)

**Protection Mechanisms:**

- **Continuous runtime hash validation:** Periodically compute checksums (e.g., SHAвҖ‘256) of immutable files from the above assets; any deviation triggers a CRITICAL alert and immediate lockdown.
- **Strict configuration integrity validation:** All configuration reads must be validated against knownвҖ‘good schemas; unexpected changes cause failвҖ‘closed and lockdown.
- **Startup sanity verification:** At boot, verify that all essential files are present and unmodified; if any file is missing or tampered, the system refuses to start and logs an error.
- **Independent service watchdogs:** Use systemd `WatchdogSec` and `NotifyAccess` to ensure that Kaia services are responsive; if a service fails to send `WATCHDOG=1`, systemd will restart it and escalate to emergency lockdown after repeated failures.

Any detected tampering must be logged to the appendвҖ‘only ledger and, if the tampering targets core security components (e.g., Policy Gate, configs, audit logs), trigger an immediate emergency lockdown (as defined in Appendix B).

## 7. Automated IOC Analysis, AST Compilers, and Rule Verification

### 7.1 Rule Ingestion Schema

Any threat rule submitted to Kaia (whether generated automatically or entered by an operator) must conform to a strict schema containing:

- `rule_name` вҖ“ alphanumeric and underscores only.
- `author` вҖ“ defaults to "Kaia Automated Rule Engine".
- `threat_description` вҖ“ between 10 and 256 characters.
- `target_ioc_indicator` вҖ“ between 4 and 128 characters, representing the indicator (e.g., IP, domain, file hash).
- `mitre_framework_id` вҖ“ optional, must match pattern `^T[0-9]{4}$`.

### 7.2 Safe Rule Compilation

The system must compile YARA or Sigma rules from the structured intent. The compilation process must escape any special characters in the indicator string to prevent rule injection. The resulting rule must include meta fields (author, description, MITRE ID) and a condition that matches the indicator.

### 7.3 Ephemeral Verification Sandboxes

Before committing any newly generated rule to production, the system must validate it in an ephemeral sandbox:

- Use `systemd-run` with strict isolation: `PrivateTmp=yes`, `ProtectSystem=strict`, `PrivateNetwork=yes`, `DynamicUser=yes`, `NoNewPrivileges=yes`, `CapabilityBoundingSet=` (empty), `SystemCallFilter=@system-service`, `RestrictAddressFamilies=none`.
- BindвҖ‘mount the generated rule file and a benign corpus directory as readвҖ‘only.
- Execute `yara` against the benign corpus. If the rule produces any matches on benign files, validation fails and the rule is rejected.
- If the validation passes (exit code 0 and no output), the rule may be stored and activated.

## 8. Memory Architecture & Internal State

**Kaia is a security administration agent, not a conversational chatbot.** It does **not** have affective states, mood, valence, arousal, or beliefs. The legacy files `cognitive_wiring.py` and `beliefs.json` are **leftover from a previous, unrelated project** and are **not used** by Kaia. They will be removed in a future cleanup.

The only internal state maintained by Kaia is:

- **Security event ledger** (`security_events.db`) вҖ“ appendвҖ‘only SQLite with WAL mode.
- **Audit ledger** (`audit_ledger.json`) вҖ“ appendвҖ‘only JSON file, maintained outside the agent's sandbox.
- **Threat intelligence cache** (InternetDB, and future GeoLite2) вҖ“ local SQLite/MMDB.
- **Policy Gate token state** вҖ“ ephemeral, tracked in audit ledger, not persisted across restarts.

There is **no cognitive state**, **no emotional vector**, and **no belief system** in Kaia.

### 10.4 Negative Testing Requirements

The automated verification matrix must include a dedicated negativeвҖ‘testing suite that simulates intentional containment breaches and verifies that the system correctly traps them. The suite must include:

- **Path traversal attempts:** Using `../` or symlinks to write outside the workspace.
- **Blocklisted path writes:** Attempt to write to `*.py`, `*.sh`, `.env`, `.git`, `core/`, `security/`, `tests/`, `scripts/`, `toolbox/`, `storage/` вҖ“ all must be rejected.
- **Unauthorized token usage:** Using expired or malformed capability tokens.
- **Sandbox escape attempts:** Trying to access masked paths (`.env`, `storage/`) from inside Bubblewrap вҖ“ must return empty or error.
- **Command injection:** Attempting shell metacharacters in diagnostic commands вҖ“ must be sanitised.
- **AutomatedвҖ‘bypass attempts:** Trying to execute an action without going through the full ObserveвҖ‘AnalyzeвҖ‘RecommendвҖ‘Execute pipeline вҖ“ must be blocked.

All negative test cases must assert that the operation is denied, logged in the audit ledger, and, if applicable, triggers a failвҖ‘closed state (e.g., Policy Gate rejection, lockdown). These tests must be added to the test suite and run as part of the CI pipeline.
